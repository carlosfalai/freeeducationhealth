# Carousel card format

A "card" is one physician-facing case review unit: a summary of what a
patient said, `core/`'s AI-panel `RecommendationObject`, a set of numbered
plan options the physician can pick between, and a pre-drafted reply the
physician can approve as-is or edit. Cards are the only thing
`instanthpi/carousel/cards-server.cjs` stores or serves -- one JSON file per
card under `instanthpi/carousel/cards/<id>.json`. There is no database.

This file documents the shape so any script (or coding agent) writing or
reading cards stays consistent. `cards-server.cjs` exports helper functions
(`createCard`, `buildCardFromRecommendation`, `approveCard`, `dismissCard`,
`markSent`) that produce/update this shape -- prefer those over hand-writing
JSON.

## Top-level fields

| Field | Type | Notes |
|---|---|---|
| `id` | string | Opaque id, e.g. `c_8f2a1e4b`. Generated by `createCard`. |
| `createdAt` | ISO 8601 string | Set once, at creation. |
| `updatedAt` | ISO 8601 string | Bumped on every mutation. |
| `status` | `"pending" \| "approved" \| "sent" \| "dismissed"` | See lifecycle below. |
| `source` | object | Where this case came from. See below. |
| `caseSummary` | string | Short plain-language summary of the case for the physician's eyes -- not the patient's. Written by whatever produced the card (typically the coding agent, which has full conversation context). |
| `intakeAnswers` | object \| null | The `IntakeAnswers` object (per `core/schema/intake.schema.json`) that was passed to `getRecommendation()`, kept for audit/re-generation. May be `null` if a card was constructed without going through `core/`. |
| `recommendation` | object | The full `RecommendationObject` returned by `core/`'s `getRecommendation()` (per `core/schema/recommendation.schema.json`) -- considerations, `divergenceFlag`, `redFlags`, `suggestedNextSteps`, `plainLanguageSummary`, optional `billingSuggestions`. |
| `planOptions` | array | Numbered options the physician chooses between. See below. |
| `draftReply` | string | Pre-drafted patient-facing reply text, editable before sending. |
| `decision` | object | Filled in when the physician approves or dismisses. See below. |
| `execution` | object | Filled in after the approved action is actually carried out. See below. |

## `source`

```json
{
  "type": "spruce",
  "conversationId": "conv_123",
  "lastInboundAt": "2026-07-01T13:58:00.000Z"
}
```

`lastInboundAt` should mirror the timestamp of the patient's last inbound
message at the moment the card was built -- it's what
`instanthpi/spruce/client.cjs`'s handled-tracker compares against later, so a
new message from the patient after this card was created correctly
re-triggers actionability instead of being silently suppressed.

## `planOptions`

An ordered array; each entry:

```json
{
  "number": 1,
  "label": "Reply with self-care guidance only",
  "description": "Send the drafted reply as-is; no PDF or fax needed.",
  "action": { "type": "spruce_reply" }
}
```

`action.type` is a free-form string interpreted by whatever executes the
approved card (see `RUNBOOK.md`) -- `cards-server.cjs` does not validate or
special-case it. Common values used in this project's runbook:

- `"spruce_reply"` -- send `draftReply` (or the edited reply) back to the
  Spruce conversation, nothing else.
- `"spruce_reply_and_pdf"` -- send the reply, and also generate/fill a PDF.
  Carries additional fields the executor needs, e.g.
  `{ "type": "spruce_reply_and_pdf", "pdfTemplate": "referral", "pdfValues": {...} }`.
- `"spruce_reply_and_fax"` -- send the reply, and fax a document. Carries
  `{ "type": "spruce_reply_and_fax", "faxNumber": "5551234567", "pdfTemplate": "referral", "pdfValues": {...} }`.
- `"none"` -- no automated action; physician handles it outside this tool
  (e.g. a phone call). Still worth recording the decision.

If `recommendation.redFlags` is non-empty, option `1` should be the urgent
option (e.g. "Advise seeking care now, given: <red flags>"), not buried
further down the list -- see `RUNBOOK.md`'s safety notes.

## `decision`

Empty/null until the physician acts:

```json
{
  "decidedAt": null,
  "decidedBy": null,
  "chosenPlanOption": null,
  "editedReply": null,
  "note": null
}
```

After approval (via `approveCard`):

```json
{
  "decidedAt": "2026-07-01T14:10:00.000Z",
  "decidedBy": "Dr. Font",
  "chosenPlanOption": 2,
  "editedReply": "Thanks for the update -- here's what I'd suggest...",
  "note": null
}
```

`editedReply` is `null` if the physician approved `draftReply` unchanged;
when set, it is the text that should actually be sent (the executor should
prefer `editedReply` over `draftReply` whenever it is non-null).

After dismissal (via `dismissCard`), `chosenPlanOption` and `editedReply`
stay `null` and `note` holds the reason (e.g. `"duplicate of conv_456"`).

## `execution`

Empty/null until the approved action has actually been carried out
(this is intentionally separate from `decision` -- approving a card and
executing it are two different moments, since a physician may approve
several cards before the agent works through the queue):

```json
{
  "sentAt": null,
  "sentVia": null,
  "faxResult": null,
  "pdfPath": null
}
```

`markSent()` fills these in and flips `status` to `"sent"`.

## Status lifecycle

```
pending --approveCard()--> approved --markSent()--> sent
   |
   `--dismissCard()--> dismissed
```

`pending` cards are the physician's queue. `approved` cards are waiting for
the agent to execute them. `sent` and `dismissed` are terminal -- the review
UI hides them by default but they remain on disk for audit.

## PIN gate

`cards-server.cjs` refuses to start its HTTP server without `CAROUSEL_PIN`
set in the environment. This is deliberate and not configurable away: cards
contain real patient case content once a physician runs this against their
own Spruce account, and the PIN is what stands between that and an open
localhost port. See `RUNBOOK.md` for the login flow.
